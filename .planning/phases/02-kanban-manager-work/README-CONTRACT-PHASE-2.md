## Открытые долги (3 шт.)

### Долг D-B2: Парсер телефонов — 2 из 5 паттернов всё ещё урезаются

**Почему:** Regex в `phone_parser.py` использует 3 альтернативы, но ни одна не покрывает:
1. **11 цифр подряд без разделителей** (`892100200475`) — regex требует группы по 2-4 цифры с разделителями, сплошной поток цифр обрезается до 10
2. **5-значный код региона** (`+7 (81146) 2-13-37`) — первая альтернатива ждёт `\d{3}` после кода страны, не `\d{5}`

Существующие 662 контакта в БД всё ещё содержат урезанные номера (например `+7 (811) 523-23` вместо `+7 (811) 523-23-36`). Cleanup-скрипт не создан.

**Когда закроется:** (a) отдельная cleanup-правка перед Фазой 3. Фикс:
- Regex: добавить альтернативу для сплошных 11 цифр: `(?:\+?7|8)\d{10}` и для 5-значных кодов: `(?:\+?7|8)[\s\-()]*\d{5}[\s\-()]*\d[\s\-()]*\d[\s\-()]*\d[\s\-()]*\d`
- Или проще: жадный паттерн `r'(?:\+?7|8)?[\s\-()]*\d{1,5}(?:[\s\-()]*\d{1,3}){2,4}'`
- Cleanup: перечитать xlsx, перепарсить телефоны, обновить Contact-записи

**Блокирует Фазу 3:** Нет — документы (КП/договор/счёт) используют реквизиты компании, не телефоны. Но менеджерам нужны полные номера для звонков.

### Долг D-E: Вложенная `<form>` в lead_info_form.html

**Почему:** В `partials/lead_info_form.html` строка 71 — `<form hx-post="/leads/{id}/assign">` вложена внутри внешней `<form hx-post="/leads/{id}/edit">` (строка 3). HTML-спецификация запрещает вложённые `<form>` — браузер может закрывать внешнюю форму раньше времени, ломая валидацию и submit.

**Когда закроется:** (b) попутно в Фазе 3 — при доработке карточки лида. Фикс: вынести форму назначения менеджера за пределы формы редактирования (отдельный `<div>` рядом). Безопасно, т.к. обе формы работают через HTMX независимо.

**Блокирует Фазу 3:** Нет. Назначение менеджера может работать нестабильно в части браузеров, но функционально роут принимает запрос правильно.

### Долг D-F: scripts/cleanup_data.py не создан

**Почему:** PLAN требовал standalone-скрипт для фикса существующих данных в БД (ИНН `.0` и аномальные level). Кодер применил фиксы в `import_service.py` для новых импортов, и данные уже оказались чистыми (вероятно, кодер пересоздал БД и переимпортировал). Но скрипт для cleanup без переимпорта — не создан.

**Когда закроется:** (c) Owner-decision «принять как known-limitation» — данные в БД уже чистые (0 ИНН с `.0`, 0 аномальных level). Скрипт нужен только если в будущем импортируются грязные данные без нормализации — но `_str_or_none` теперь обрабатывает это. Долг не нужен, закрываю.

**Блокирует Фазу 3:** Нет.

## Архитектурные замечания (информационные)

1. **Канбан рендерит все лиды одним запросом** (592 карточки → 371KB HTML). Для текущего объёма (~600 лидов) это приемлемо. При росте до 2000+ — потребуется пагинация или lazy-load колонок. Не блокер.

2. **`api_change_stage` возвращает JSON, а не HTML-фрагмент** — отличие от PLAN (ожидался HTML-свап колонок). Но kanban.js написан под JSON-ответ: при успехе обновляет счётчики на клиенте, при 422 возвращает карточку. Это работающее решение, чуть другой подход к HTMX-паттерну.

3. **`is_overdue` — динамический атрибут** на Task, устанавливается в роуте (не в модели). Правильный подход — это presentational-логика, не должна быть в ORM-модели.

4. **`tasks.py` роут `update_task_status`** возвращает `partials/task_row.html` — но `hx-target="closest div"` в select'е. Работает, т.к. task_row — корневой `<div>`. ОК.

5. **Сортировка ContactLog в шаблоне** через Jinja2 `|sort(attribute='contact_date', reverse=true)` — работает, но при большом количестве записей лучше сортировать в SQL. Не блокер.

6. **Кодер НЕ создал `app/routes/contacts.py`, `app/routes/comments.py`** — PLAN предлагал их как "НЕ нужен", кодер правильно объединил всё в `leads.py`. Хорошее решение.
